Assignment 2 implemented the ETL pipeline as a single monolithic class (org.howard.edu.lsp.assignment2.ETLPipeline) containing an inner Product type and static methods for extraction, transformation, and loading. All logic—including parsing, validation, CSV I/O, row counting, and transformation—was combined in one file, which made the code functional but difficult to extend or test. For Assignment 3, I redesigned the solution to use only three public classes—Product, CsvHandler, and ETLPipeline—while preserving exactly the same behavior, inputs, outputs, and transformation rules as A2.
The three classes each have a focused responsibility. Product represents a single product record, encapsulating its fields with private access and getters/setters. CsvHandler handles CSV reading, writing, and parsing, including a parseRowToProduct helper that enforces A2’s rules: exactly four columns, numeric ID and price, and preservation of trailing empty fields with split(",", -1). ETLPipeline orchestrates the pipeline, performing extraction, transformation, and loading. It retains row counters (rowsRead, rowsTransformed, rowsSkipped), reproduces header detection, blank-line stopping logic, and applies all original transformation rules, including uppercasing names, a 10% discount for Electronics, rounding with HALF_UP to two decimals, Premium Electronics assignment for discounted prices above 500, and consistent price-range thresholds.
This redesign separates concerns into distinct classes, increasing modularity, readability, and maintainability while respecting the three-class constraint. Object-oriented principles were applied carefully: objects represent domain data (Product instances), classes organize responsibilities, and encapsulation protects internal state in both data and processing logic. No additional OO concepts like inheritance or polymorphism were introduced to remain within the assignment’s three-concept limit, but the design is extensible for future interfaces or transformations without altering the public API.
To ensure A3 behaved identically to A2, I conducted several tests. Normal input produced identical outputs, verified using diff -u. An empty input file resulted in only the header line, and a missing input file triggered a graceful error, both matching A2 behavior. Malformed rows—wrong number of columns or non-numeric prices—were skipped with rowsSkipped incremented, as in A2. Edge cases involving rounding and Premium assignment confirmed that transformations were applied exactly the same. All row counters were confirmed to match A2 for consistency.
A generative AI assistant supported the design process by suggesting minimal OO decompositions, providing code snippets for CSV parsing and BigDecimal rounding, and drafting Javadoc templates. All AI-generated suggestions were carefully reviewed and adapted to ensure that A3 exactly reproduced A2’s functionality.
In conclusion, Assignment 3 preserves the exact behavior and outputs of Assignment 2 while introducing a clear, modular object-oriented design using only object, class, and encapsulation. The resulting code is easier to understand, test, and maintain, and the design process demonstrates both disciplined OO application and thoughtful adaptation of AI-assisted suggestions.
